S-expression 인터프리터

S-expression이란?

괄호로 감싼 리스트 구조로 코드와 데이터를 표현하는 표기법이다. Lisp 계열 언어에서 대표적으로 사용된다.

Lisp 계열의 철학

프로그램 자체도 리스트 데이터처럼 다룰 수 있다는 점이 핵심이다. 이 철학 덕분에 함수형 사고, 재귀, 메타프로그래밍, 언어 확장이 자연스럽게 이어진다.

왜 이런 문법이 생겼나?

1950년대 후반 John McCarthy가 인공지능 연구에서 기호와 리스트를 다루기 위해 Lisp를 설계하면서 등장했다. 사람이 읽기 쉬운 문법보다, 코드와 데이터를 같은 구조로 다룰 수 있는 단순한 표현력을 우선했다.

어떤 특징인가?

연산자와 피연산자가 같은 리스트 안에 놓여 문법 구조가 단순하다. 파싱 결과도 트리 형태로 자연스럽게 이어진다.

prompt session 
>>> func fib (n)
...   (if (< n 2)
...     (return n)
...     (return (+ (fib (- n 1)) (fib (- n 2)))))
...
>>> print (fib 10)
55.0

Team Decision

언어 수정사항

S-expression은 모든 표현을 소괄호로 감싸주어야 한다. 우리는 REPL에서 더 자연스럽게 코드를 입력하도록, 바깥쪽 global 소괄호를 제거하기로 했다.

Before

((var total 0)
(set! total (+ total 1))
(print total))

After

var total 0
set! total (+ total 1)
print total

Best Practices

Best Practices 사례

기능을 완성하는 것만큼 중요했던 것은, 새 코드가 들어와도 이전 기능을 믿을 수 있고 문제를 빠르게 발견할 수 있는 협업 구조였다.

1. GitHub Actions로 회귀 테스트 강제

기능 개발이 끝날 때마다 해당 테스트를 회귀 테스트 목록에 넣고, PR에서 pytest가 100% 통과하지 않으면 merge할 수 없도록 했다.

효과: 신규 기능이 기존 기능을 깨뜨리는지 자동으로 확인했고, 리뷰어가 수동으로 전체 시나리오를 다시 실행하지 않아도 안정성을 판단할 수 있었다.

2. 기능 완성 후 사용자 관점 테스트

구현이 끝난 뒤 실제 사용 흐름으로 입력해보며 내부 테스트만으로는 드러나기 어려운 버그를 찾아냈다.

  • 선언 실패 후 재선언 시 이미 정의된 변수로 처리되던 문제
  • if 조건문에 void 함수를 넣으며 null 미구현을 발견한 사례
  • 함수 정의 시 함수명이 자동 출력되던 문제

효과: 테스트가 “정해진 API”를 지킨다면, 사용자 테스트는 “실제로 쓰기 편한지”를 확인해주는 마지막 안전망이 되었다.

3. 패턴과 구조를 확장성 관점에서 점검

과제에서 제시된 처리 단계를 기반으로 Pipeline 구조와 Interpreter 구조를 자연스럽게 갖추었고, 추가 기능 구현 과정에서 Dispatch Table이 확장성을 높이는 핵심 장치가 되었다.

Pipeline 처리 단계 분리 Interpreter AST 실행 모델 Dispatch 문법 확장 지점 정리 Visitor AST 순회 리팩토링 후보 Command debug 명령 리팩토링 후보

효과: 현재 구조의 강점과 다음 리팩토링 후보를 구분해, 발표 후에도 개선 방향을 설명할 수 있게 되었다.

Test Driven Collaboration

TDD는 테스트 기술이 아니라 협업 방식이었다

테스트는 모듈 담당자 사이의 계약이 되었다. 먼저 사용법과 기대 동작을 테스트로 정하니 분업 범위가 선명해졌고, 진척률도 감이 아니라 통과 여부로 확인할 수 있었다.

  • 기능 구현 전 RED 테스트로 API 사용법 공유
  • 유닛 테스트로 모듈 책임 고정
  • 통합 테스트로 사용자 시나리오 보호
  • 리팩토링 후 전체 테스트로 회귀 확인

소감

김일도님

AI 도구가 프로젝트 이해, 리팩토링, 테스트 보강까지 도와줄 수 있다는 점이 인상적이었다. 현업에서도 오래된 코드를 개선하는 데 적극 활용해보고 싶다.

손지영님

이슈와 버그를 함께 해결하려는 분위기가 좋았다. 중간중간 리팩토링을 하니 구조 이해와 기능 추가가 쉬워졌고, 나중에 한 번에 고치는 것보다 비용이 적다는 것을 체감했다.

윤길중님

TDD와 GitHub 기반 협업 프로세스를 직접 경험한 의미 있는 시간이었다. AI 도구의 강력함도 확인했고, 보안 기준이 갖춰진다면 부서 DX에도 활용 가능성이 크다고 느꼈다.

여상훈님

GitHub로 함께 개발하니 같은 목표를 향해 움직인다는 느낌이 컸다. PR rule로 안정성을 확보할 수 있었고, AI 도구도 기대보다 편하고 강력했다.

김상민님

TDD가 매우 강력하다고 느꼈다. 사용 방식과 기대 동작을 먼저 정하고 개발하니 목표가 흔들리지 않았고, 진척률도 테스트 통과 여부로 자연스럽게 확인할 수 있었다.
