돌아가는 코드에서 함께 바꿀 수 있는 코드로

같은 인터프리터 과제 안에서 우리 팀은 사용성 선택, 기능 신뢰성, 리팩토링, TDD 협업을 하나의 흐름으로 연결했다.

prompt session

Team Decision

우리 팀의 언어 사용성 선택

S-expression은 공통 과제의 기반이었지만, 사용자 입력 경험은 팀에서 다시 판단했다. 우리는 바깥쪽 global 소괄호를 제거해 REPL에서 더 자연스럽게 코드를 입력하도록 만들었다.

Before

((var total 0)
(set! total (+ total 1))
(print total))

After

var total 0
set! total (+ total 1)
print total

Reliability First

기능 데모로 먼저 신뢰를 확보한다

모든 기능을 나열하기보다, 발표에서는 대표 흐름 하나로 충분하다. 코드 입력, 파일 실행, 오류 보고, 디버깅을 연결해 실제로 사용할 수 있는 언어라는 신뢰를 먼저 만든다.

1 Prompt

사용자가 한 줄씩 입력하고 세션 상태를 유지한다.

2 File

소스 파일을 읽고 오류 줄 번호를 포함해 실행 결과를 보여준다.

3 Debug

Stmt 단위 step, breakpoint, continue로 실행 흐름을 확인한다.

4 Watch

중단 지점마다 변수 값을 확인해 상태 변화를 추적한다.

Before & After

리팩토링은 변경 비용을 낮추는 작업이었다

발표에서는 세 가지 예시만 보여준다. 핵심은 코드가 예뻐졌다는 말이 아니라, 다음 기능을 더 작고 안전하게 추가할 수 있게 되었다는 점이다.

1. Environment 캡슐화

디버그 watch가 실행 환경 내부 딕셔너리에 직접 접근하던 구조를 `lookup_variable`, `current_scope_snapshot` 같은 공개 API로 바꿨다. 

# Before
self._executor._environment._values
# After
self._executor.current_scope_snapshot()
self._executor.lookup_variable(name)

2. 배열 연산 검증 분리

`Array`, `index`, `set-index!`마다 반복되던 숫자/범위 검증을 헬퍼로 분리해 오류 정책을 한 곳에서 관리하게 했다. 

# Before
if not isinstance(index, (int, float)):
    raise ExecuteError(...)
if idx < 0 or idx >= len(array):
    raise ExecuteError(...)
# After
idx = self._as_array_index(array, value)
size = self._as_array_size(value)

3. 확장 포인트를 테이블로 정리

새로운 문법과 실행 노드가 추가될 때 조건문이 퍼지지 않도록 Assembler와 Executor의 dispatch table을 기준으로 확장했다. 

# Before
if token == "var": ...
elif token == "print": ...
elif token == "if": ...
# After
_SPECIAL_FORMS = {
  TokenType.VAR: "_parse_var_stmt",
  TokenType.PRINT: "_parse_print_stmt",
}

Test Driven Collaboration

TDD는 테스트 기술이 아니라 협업 방식이었다

테스트는 모듈 담당자 사이의 계약이 되었다. 먼저 사용법과 기대 동작을 테스트로 정하니 분업 범위가 선명해졌고, 진척률도 감이 아니라 통과 여부로 확인할 수 있었다.

  • 기능 구현 전 RED 테스트로 API 사용법 공유
  • 유닛 테스트로 모듈 책임 고정
  • 통합 테스트로 사용자 시나리오 보호
  • 리팩토링 후 전체 테스트로 회귀 확인

What We Learned

현업에 가져갈 수 있는 배움

인터페이스를 먼저 정한다

함수 이름, 입력, 출력, 실패 조건이 먼저 정해지면 각자 구현해도 합류 지점이 흔들리지 않는다.

작은 PR은 리뷰 품질을 올린다

stacked PR로 기능 단위를 작게 유지하니 리뷰어가 맥락을 잃지 않고 피드백할 수 있었다.

리팩토링은 협업 비용을 줄인다

캡슐화와 책임 분리는 단순 미화가 아니라 다음 변경을 더 예측 가능하게 만드는 투자였다.
