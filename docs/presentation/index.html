S-expression 인터프리터

S-expression이란?

괄호로 감싼 리스트 구조로 코드와 데이터를 표현하는 표기법이다. Lisp 계열 언어에서 대표적으로 사용된다.

어떤 특징인가?

연산자와 피연산자가 같은 리스트 안에 놓여 문법 구조가 단순하다. 파싱 결과도 트리 형태로 자연스럽게 이어진다.

왜 만들었나?

언어 처리의 핵심 단계인 토큰화, 조립, 검사, 실행을 직접 구현해보기 위해 개발했다. 우리는 그 위에 테스트와 리팩토링을 얹어 협업 가능한 코드로 다듬었다.

prompt session 
>>> var total 0
... 
>>> set! total (+ total 3)
... 
>>> print total
3.0

debug> watch total
Watch total: 3.0

Team Decision

우리 팀의 언어 사용성 선택

S-expression은 공통 과제의 기반이었지만, 사용자 입력 경험은 팀에서 다시 판단했다. 우리는 바깥쪽 global 소괄호를 제거해 REPL에서 더 자연스럽게 코드를 입력하도록 만들었다.

Before

((var total 0)
(set! total (+ total 1))
(print total))

After

var total 0
set! total (+ total 1)
print total

Reliability First

입력과 결과를 함께 보여준다

발표 중 전체 데모를 직접 실행하기 어렵다면, 사용자가 입력하는 코드와 인터프리터가 돌려주는 결과를 나란히 보여주는 방식이 가장 선명하다.

Before & After

디자인 패턴으로 확장 지점을 만들었다

리팩토링의 핵심은 단순 정리가 아니라 새 기능을 어디에 추가할지 예측 가능하게 만드는 것이었다. 발표에서는 패턴이 코드 품질을 올린 대표 사례 세 가지만 보여준다.

1. Factory Method 관점의 실행 모드 분리

프롬프트, 파일, 디버그 실행을 한 흐름에 섞지 않고 mode별 생성 함수로 분리했다. 새 실행 모드가 생겨도 기존 모드의 입출력 흐름을 건드리지 않아도 된다. 

# Before
if mode == "prompt":
    ...
elif mode == "run":
    ...
elif mode == "debug":
    ...
# After
run_prompt_mode(...)
run_file_mode(path)
run_debug_mode(source, commands)

2. Dispatch Table로 문법 확장 비용 축소

Assembler와 Executor에 분기문이 계속 늘어나던 구조를 테이블 기반 dispatch로 정리했다. 문법이나 Stmt가 추가될 때 등록 지점이 한 곳으로 모인다. 

# Before
if token == "var": ...
elif token == "print": ...
elif token == "class": ...
elif token == "import": ...
# After
_SPECIAL_FORMS = {
  TokenType.VAR: "_parse_var_stmt",
  TokenType.CLASS: "_parse_class_stmt",
  TokenType.IMPORT: "_parse_import_stmt",
}

_STMT_DISPATCH = {
  ClassStmt: "_execute_classstmt",
  ImportStmt: "_execute_importstmt",
}

3. Template Method로 오류 보고 정책 확장

기본 Executor의 실행 알고리즘은 유지하고, 파일 모드에서 필요한 줄 번호 기반 런타임 오류 보고만 하위 클래스에서 확장했다. 

# Before
def execute(program):
    for stmt in program.statements:
        try:
            execute_stmt(stmt)
        except ExecuteError:
            print("Runtime error")
# After
class FileModeExecutor(SExpressionExecutor):
    def _execute_stmt(self, stmt):
        try:
            return super()._execute_stmt(stmt)
        except ExecuteError as error:
            raise RuntimeExecutionError(
                stmt.location.line, error
            )

Test Driven Collaboration

TDD는 테스트 기술이 아니라 협업 방식이었다

테스트는 모듈 담당자 사이의 계약이 되었다. 먼저 사용법과 기대 동작을 테스트로 정하니 분업 범위가 선명해졌고, 진척률도 감이 아니라 통과 여부로 확인할 수 있었다.

  • 기능 구현 전 RED 테스트로 API 사용법 공유
  • 유닛 테스트로 모듈 책임 고정
  • 통합 테스트로 사용자 시나리오 보호
  • 리팩토링 후 전체 테스트로 회귀 확인

What We Learned

현업에 가져갈 수 있는 배움

인터페이스를 먼저 정한다

함수 이름, 입력, 출력, 실패 조건이 먼저 정해지면 각자 구현해도 합류 지점이 흔들리지 않는다.

작은 PR은 리뷰 품질을 올린다

stacked PR로 기능 단위를 작게 유지하니 리뷰어가 맥락을 잃지 않고 피드백할 수 있었다.

리팩토링은 협업 비용을 줄인다

캡슐화와 책임 분리는 단순 미화가 아니라 다음 변경을 더 예측 가능하게 만드는 투자였다.
