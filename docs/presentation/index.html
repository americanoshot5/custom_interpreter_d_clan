S-expression 인터프리터

S-expression이란?

괄호로 감싼 리스트 구조로 코드와 데이터를 표현하는 표기법이다. Lisp 계열 언어에서 대표적으로 사용된다.

Lisp 계열의 철학

프로그램 자체도 리스트 데이터처럼 다룰 수 있다는 점이 핵심이다. 이 철학 덕분에 함수형 사고, 재귀, 메타프로그래밍, 언어 확장이 자연스럽게 이어진다.

왜 이런 문법이 생겼나?

1950년대 후반 John McCarthy가 인공지능 연구에서 기호와 리스트를 다루기 위해 Lisp를 설계하면서 등장했다. 사람이 읽기 쉬운 문법보다, 코드와 데이터를 같은 구조로 다룰 수 있는 단순한 표현력을 우선했다.

어떤 특징인가?

연산자와 피연산자가 같은 리스트 안에 놓여 문법 구조가 단순하다. 파싱 결과도 트리 형태로 자연스럽게 이어진다.

prompt session 
>>> func fib (n)
...   (if (< n 2)
...     (return n)
...     (return (+ (fib (- n 1)) (fib (- n 2)))))
...
>>> print (fib 10)
55.0

Before & After

디자인 패턴 적용 사례

패턴의 목적은 이름을 붙이는 것이 아니라 변경 영향 범위를 줄이는 것이다. 아래 세 사례는 기능 추가 시 수정 위치를 제한해 가독성과 확장성을 높인 리팩토링이다.

1. Pipeline 구조로 처리 단계 분리

소스 코드를 한 번에 실행하지 않고 Tokenizer, Assembler, Checker, Optimizer, Executor가 순서대로 Program을 넘겨받는 구조로 나눴다. 단계별 책임이 분리되어 중간 처리 단계를 추가하기 쉬웠다. 

# Before
def run(source):
    tokens = tokenize(source)
    program = assemble(tokens)
    check(program)
    execute(program)

# 최적화 추가 시 실행 흐름마다
# 중복 수정이 필요해지기 쉽다.
# After
tokens = tokenize(source)
program = assemble(tokens)
check(program)

optimized = optimize(program)
check(optimized)
execute(optimized)

변경 영향: optimizer가 새로 들어와도 앞단의 토큰화/조립 로직과 뒷단의 실행 로직을 직접 흔들지 않고, `Program -> Program` stage로 제한해서 붙일 수 있었다.

2. Dispatch Table로 문법 확장 비용 축소

Assembler와 Executor에 분기문이 계속 늘어나던 구조를 테이블 기반 dispatch로 정리했다. 문법이나 Stmt가 추가될 때 등록 지점이 한 곳으로 모인다. 

# Before
if token == "var": ...
elif token == "print": ...
elif token == "class": ...
elif token == "import": ...
# After
_SPECIAL_FORMS = {
  TokenType.VAR: "_parse_var_stmt",
  TokenType.CLASS: "_parse_class_stmt",
  TokenType.IMPORT: "_parse_import_stmt",
}

_STMT_DISPATCH = {
  ClassStmt: "_execute_classstmt",
  ImportStmt: "_execute_importstmt",
}

변경 영향: `class`, `import`, `return` 같은 문법을 추가할 때 긴 조건문을 수정하지 않고 테이블 등록과 전용 메서드 추가로 끝낼 수 있었다.

3. Template Method로 오류 보고 정책 확장

기본 Executor의 실행 알고리즘은 유지하고, 파일 모드에서 필요한 줄 번호 기반 런타임 오류 보고만 하위 클래스에서 확장했다. 

# Before
def execute(program):
    for stmt in program.statements:
        try:
            execute_stmt(stmt)
        except ExecuteError:
            print("Runtime error")
# After
class FileModeExecutor(SExpressionExecutor):
    def _execute_stmt(self, stmt):
        try:
            return super()._execute_stmt(stmt)
        except ExecuteError as error:
            raise RuntimeExecutionError(
                stmt.location.line, error
            )

변경 영향: 기본 실행기의 알고리즘은 유지하고 파일 모드의 오류 표시 정책만 확장해, 공통 실행 동작의 회귀 위험을 줄였다.

Test Driven Collaboration

TDD는 테스트 기술이 아니라 협업 방식이었다

테스트는 모듈 담당자 사이의 계약이 되었다. 먼저 사용법과 기대 동작을 테스트로 정하니 분업 범위가 선명해졌고, 진척률도 감이 아니라 통과 여부로 확인할 수 있었다.

  • 기능 구현 전 RED 테스트로 API 사용법 공유
  • 유닛 테스트로 모듈 책임 고정
  • 통합 테스트로 사용자 시나리오 보호
  • 리팩토링 후 전체 테스트로 회귀 확인

What We Learned

현업에 가져갈 수 있는 배움

인터페이스를 먼저 정한다

함수 이름, 입력, 출력, 실패 조건이 먼저 정해지면 각자 구현해도 합류 지점이 흔들리지 않는다.

작은 PR은 리뷰 품질을 올린다

stacked PR로 기능 단위를 작게 유지하니 리뷰어가 맥락을 잃지 않고 피드백할 수 있었다.

리팩토링은 협업 비용을 줄인다

캡슐화와 책임 분리는 단순 미화가 아니라 다음 변경을 더 예측 가능하게 만드는 투자였다.
