S-expression 인터프리터

S-expression이란?

괄호로 감싼 리스트 구조로 코드와 데이터를 표현하는 표기법이다. Lisp 계열 언어에서 대표적으로 사용된다.

Lisp 계열의 철학

프로그램 자체도 리스트 데이터처럼 다룰 수 있다는 점이 핵심이다. 이 철학 덕분에 함수형 사고, 재귀, 메타프로그래밍, 언어 확장이 자연스럽게 이어진다.

왜 이런 문법이 생겼나?

1950년대 후반 John McCarthy가 인공지능 연구에서 기호와 리스트를 다루기 위해 Lisp를 설계하면서 등장했다. 사람이 읽기 쉬운 문법보다, 코드와 데이터를 같은 구조로 다룰 수 있는 단순한 표현력을 우선했다.

어떤 특징인가?

연산자와 피연산자가 같은 리스트 안에 놓여 문법 구조가 단순하다. 파싱 결과도 트리 형태로 자연스럽게 이어진다.

prompt session 
>>> func fib (n)
...   (if (< n 2)
...     (return n)
...     (return (+ (fib (- n 1)) (fib (- n 2)))))
...
>>> print (fib 10)
55.0

Team Decision

언어 수정사항

S-expression은 모든 표현을 소괄호로 감싸주어야 한다. 우리는 REPL에서 더 자연스럽게 코드를 입력하도록, 바깥쪽 global 소괄호를 제거하기로 했다.

(print total))

Best Practices

Best Practices

기능을 완성하는 것만큼 중요했던 것은, 새 코드가 들어와도 이전 기능을 믿을 수 있고 문제를 빠르게 발견할 수 있는 협업 구조였다.

1. 패턴 적용 범위와 리팩토링 의사결정

모든 패턴을 같은 깊이로 적용하지 않고, 문제의 범위에 맞는 구조를 선택했다. 적용한 패턴과 의도적으로 보류한 패턴을 함께 정리해 리팩토링 판단 기준을 남겼다.

Pipeline 프로젝트 최상위 흐름

Tokenizer → Assembler → Checker → Executor 단계로 책임을 분리했다.

Interpreter AST 실행 모델

Stmt와 Expr를 실행 단위로 삼아 언어 문법을 런타임 동작으로 연결했다.

Dispatch 문법 확장 지점

Checker와 Executor 안의 타입별 분기를 테이블로 모아 변경 범위를 제한했다.

Command Debug Shell 명령

debug 명령을 Command 객체로 분리해 명령 추가와 수정 범위를 줄였다.

Visitor 검토 후 미적용

AST 전체에 accept/visit 구조를 넣는 비용 대비 효과가 작아 보류했다.

Interpreter: AST를 실행 단위로 관리

Before 
# 문자열을 바로 실행하려는 구조
run_line("(var x 1)")

if tokens[0] == "var":
    env[name] = eval(value)
elif tokens[0] == "print":
    print(eval(expr))
After 
# 문법을 AST 객체로 먼저 표현
Program([
  VarStmt("x", LiteralExpr(1)),
  PrintStmt(IdentifierExpr("x")),
])

# 실행기는 AST를 해석만 담당
executor.execute(program)

효과: 문법 구조를 AST로 고정하고, Checker/Executor가 같은 구조를 각자 목적에 맞게 해석할 수 있었다.

Dispatch: 변경 영향도 축소

Before 
def execute(stmt):
    if isinstance(stmt, VarStmt):
        ...
    elif isinstance(stmt, IfStmt):
        ...
    elif isinstance(stmt, WhileStmt):
        ...  # 새 문법을 중간에 추가
    elif isinstance(stmt, ForStmt):
        ...
After 
_STMT_DISPATCH = {
    VarStmt: "_execute_varstmt",
    IfStmt: "_execute_ifstmt",
    ForStmt: "_execute_forstmt",
    WhileStmt: "_execute_whilestmt",
}

def _execute_whilestmt(stmt):
    ...

효과: 새 Stmt 추가 시 분기 함수 전체를 건드리는 대신, 모듈별 dispatch table 1줄과 전용 메서드로 변경 지점을 좁혔다.

Command: debug 명령 분리

Before 
def _handle_command(text):
    if text == "continue":
        self._continue()
    elif text == "inspect":
        self._inspect_scope()
    elif text.startswith("watch "):
        self._add_watch(text)
    ...
After 
# 1. 신규 명령 동작만 작성
class RestartDebugCommand:
    def execute(session):
        ...

# 2. 생성 담당 parser에 1줄 등록
if text == "restart":
    return RestartDebugCommand()

효과: 신규 명령은 자기 Command 클래스에 동작을 작성하고, 생성 담당 parser에 한 줄 등록하면 된다.

효과: 패턴을 많이 넣는 것보다, 변경이 자주 일어나는 지점에만 적용해 가독성과 확장성을 높였다.

2. GitHub Actions로 회귀 테스트 강제

기능 개발이 끝날 때마다 해당 테스트를 회귀 테스트 목록에 넣고, PR에서 pytest가 100% 통과하지 않으면 merge할 수 없도록 했다.

효과: 신규 기능이 기존 기능을 깨뜨리는지 자동으로 확인했고, 리뷰어가 수동으로 전체 시나리오를 다시 실행하지 않아도 안정성을 판단할 수 있었다.

3. 기능 완성 후 사용자 관점 테스트

구현이 끝난 뒤 실제 사용 흐름으로 입력해보며 내부 테스트만으로는 드러나기 어려운 버그를 찾아냈다.

선언 실패 후 재선언 오류

문법 오류로 실패한 선언이 환경에 남아, 올바른 재선언까지 중복 변수로 처리되던 문제를 발견했다.

(var y (10 + 1))  ; 실패
(var y (+ 10 1))  ; y가 이미 정의됨
PR #39

함수 정의 결과가 자동 출력

REPL에서 함수 정의만 했는데 함수명이 결과처럼 출력되어, 실행 결과와 선언 결과가 섞여 보이던 문제를 수정했다.

>>> func xx (n) (print n)
xx  ; 기존: 함수명이 자동 출력됨
>>> xx 20
20.0
PR #41

void 함수 조건식과 null 지원

값을 반환하지 않는 함수를 if 조건에 넣어보며, null 값이 언어 차원에서 충분히 처리되지 않는 문제를 확인했다.

(func logOnly ()
  (print "check"))
(if (logOnly)
  (print "then")
  (print "else"))
PR #42

사용자 테스트는 “실제로 쓰기 편한지”를 확인해주는 마지막 안전망이 되었다.

Test Driven Collaboration

테스트 분포

기능이 늘어나고 내부 구조가 바뀌어도, 테스트가 이전 동작을 계속 확인해주었다. 덕분에 새 기능을 붙이거나 리팩토링할 때 무엇이 깨졌는지 빠르게 알 수 있었고, 개발 속도와 안정성을 함께 얻을 수 있었다.

  • 기능별 테스트로 기대 동작을 먼저 고정
  • 변경 후 pytest로 기존 기능 회귀 여부 확인
  • 통합 테스트로 실제 사용자 흐름까지 보호
  • 리팩토링 후에도 같은 결과가 유지되는지 검증
